# Binpacker Release Prep

Follow this workflow when preparing a new `binpacker` gem release.

## Branch: `release/x.y.z`

Release work happens on a dedicated `release/x.y.z` branch, not directly on `master`. Cut it from an up-to-date `master`:

```sh
git switch master && git pull
git switch -c release/x.y.z
```

Every step below — the metadata edits, the local verify, the version-bump commit — lands on this branch.

## Update Release Metadata

Decide the next semantic version first, then update all versioned files together.

Update these files:

- `CHANGELOG.md`
- `lib/binpacker/version.rb`
- `Gemfile.lock` (regenerated by `bundle install` after the version bump)

### Seal the `[Unreleased]` entries — the mandatory rewrite step

**Do this first, before the mechanical version-heading move below.** The intended state is that every `[Unreleased]` entry is already written release-style — this step is **cross-entry consolidation**: fold several commits' entries into one user-recognisable change, reorder, dedupe, and split any merge artefacts.

The canonical rules:

- **One sentence per top-level bullet.** One period, no em-dash clauses, no run-ons.
- **Subsystem label prefix**: `**[CLI]**`, `**[scheduling]**`, `**[work-stealing]**`, `**[calibration]**`, etc.
- **User-facing only.** Cut internal implementation detail.
- **Supplementary detail goes in child items** (`  - …`), two to three sentences max.

Procedure:

1. Read the entire `[Unreleased]` block. For **each** top-level bullet, classify it: release-style (leave) or commit-style (rewrite).
2. Rewrite every commit-style bullet.
3. Watch for **merge artefacts**: two entries accidentally glued into one bullet. Split them.
4. Re-read the sealed section top-to-bottom as a user would.

### Write the release summary

Immediately under the new `## [x.y.z] - YYYY-MM-DD` heading, **before the first `###` section**, write a short prose summary (≈3–4 sentences) that gives a reader the *themes* of the release before the itemised entries.

### Release mechanics

- Add a new `## [x.y.z] - YYYY-MM-DD` section immediately below `[Unreleased]`.
- Open it with the release summary, then `### Added` / `### Changed` / `### Fixed` sections.
- Use Keep a Changelog section headings verbatim: `Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`.
- Update the `[Unreleased]` compare link and add the new release link at the bottom of `CHANGELOG.md`. Compare links target `https://github.com/rigortype/binpacker/compare/...`.
- Bump `Binpacker::VERSION` in `lib/binpacker/version.rb` to match the new release.
- Run `bundle install` after the version bump so `Gemfile.lock` records the new `binpacker (x.y.z)` revision.
- Keep the release commit message in the form `Bump up version to x.y.z`.

## Verify the Release

Run before committing:

```sh
make verify
git diff --check
```

`make verify` runs `make test` and `make lint`. `git diff --check` catches whitespace mistakes.

Also build the gem to confirm the gemspec is still valid:

```sh
gem build binpacker.gemspec
```

The build produces `binpacker-x.y.z.gem` in the working tree. Delete it before committing — built gems must not be checked in.

## Commit the Result

Prefer a single release-prep commit containing:

- the `lib/binpacker/version.rb` bump
- the `CHANGELOG.md` update for the new version
- the `Gemfile.lock` regenerated by `bundle install`

Use:

```text
Bump up version to x.y.z
```

## Push and watch CI

```sh
git push -u origin release/x.y.z
```

This triggers `ci.yml` — tests across Ruby 3.2, 3.3, and 3.4. Do not merge until the gate is green.

## Open the release PR and merge it on green

```sh
gh pr create --base master --head release/x.y.z \
  --title "Bump up version to x.y.z" --body "<short release summary>"
gh pr checks <pr> --watch
gh pr merge <pr> --rebase --delete-branch
```

Merge rebase (not squash) to keep the `Bump up version to x.y.z` commit intact.

## Publish

After the release PR merges, publish from an up-to-date `master`:

```sh
git switch master && git pull
bundle exec rake release
```

`rake release` verifies a clean working tree, tags the release as `vx.y.z`, pushes the tag to `origin`, publishes the gem to RubyGems, AND creates the matching GitHub Release (hooked via `rake release:github`). It requires:

- A clean working tree on `master`.
- A RubyGems API key with MFA configured — `binpacker.gemspec` sets `rubygems_mfa_required => "true"`.
- Push access to `origin`.
- `gh auth status` clean — the GitHub Release step shells out to `gh release create`.

If the GitHub Release step fails after the gem is already published, retry that step alone:

```sh
bundle exec rake release:github
```

## Quick Checklist

- Working tree starts clean or every pending change is understood.
- `Binpacker::VERSION` (`lib/binpacker/version.rb`) and `Gemfile.lock` agree on the new version.
- **Every** former `[Unreleased]` bullet was classified and, if commit-style, rewritten.
- The new `## [x.y.z]` section opens with a short release-summary paragraph before `### Added`.
- `[Unreleased]` / `[x.y.z]` compare links resolve.
- `make verify` passed.
- `gem build binpacker.gemspec` succeeded and the produced `.gem` is not committed.
- The final commit message follows `Bump up version to x.y.z`.
- Release work happened on a `release/x.y.z` branch (not directly on `master`).
- Before merge: `ci.yml` gate is green.
- The release PR merged to `master` keeping the `Bump up version to x.y.z` commit intact.
- After publish: the `vx.y.z` tag, the RubyGems push, and the GitHub Release all exist; the release branch is deleted.
